## 2. 项目背景 / Project Background

工业设备会持续产生温度、振动、电流等状态数据，这些数据可以直接反映设备的健康情况。
Industrial devices continuously generate status data such as temperature, vibration, and current, and these metrics can directly reflect device health.

如果这些数据不能被及时采集、分析和展示，现场人员就很难快速发现异常。
If this data cannot be collected, analyzed, and visualized in time, it becomes difficult for operators to detect anomalies quickly.

所以我设计了这个项目，希望打通从设备侧到运维看板的完整监控闭环。
So I designed this project to build a complete monitoring loop from the device side to the operations dashboard.

## 3. 一句话概括架构 / One-Sentence Architecture

这个项目可以概括为：`C++ 负责设备模拟和网关发布，MQTT 负责实时传输，Python 负责处理和落库，Streamlit 负责监控展示和运维交互。`
This project can be summarized as: `C++ handles device simulation and gateway publishing, MQTT handles real-time transport, Python handles processing and persistence, and Streamlit handles monitoring visualization and operational interaction.`

---

## 4. 数据流转 / Data Flow

首先，C++ 侧会模拟设备传感器，生成温度、振动和电流等遥测数据。
First, the C++ side simulates device sensors and generates telemetry data such as temperature, vibration, and current.

然后，网关会把这些数据序列化成 JSON，并通过 MQTT 发布到 Broker。
Then, the gateway serializes this data into JSON and publishes it to the broker through MQTT.

接着，Python 处理器订阅这些消息，对数据做标准化、健康评分和异常判定。
Next, the Python processor subscribes to these messages and performs normalization, health scoring, and anomaly detection.

处理完成后，系统会把结果写入 SQLite，并且在需要时也可以同步写入 MySQL。
After processing, the system writes the results into SQLite, and it can also mirror them into MySQL when needed.

最后，Dashboard 从数据库读取最新遥测、历史趋势、告警记录和维护日志，展示给用户。
Finally, the dashboard reads the latest telemetry, historical trends, alert records, and maintenance logs from the database and presents them to users.

所以这条链路本质上是：`设备 -> MQTT -> 后端处理 -> 数据库 -> Dashboard`。
So the essence of this pipeline is: `device -> MQTT -> backend processing -> database -> dashboard`.

## 7. 技术选型思路 / Technology Choices

我用 C++ 做设备侧模拟，是因为它更接近边缘设备和高性能采集场景。
I used C++ for the device-side simulation because it is closer to edge devices and high-performance collection scenarios.

我用 MQTT 做传输，是因为它轻量、实时、解耦，特别适合 IoT 场景。
I used MQTT for transport because it is lightweight, real-time, and decoupled, which makes it very suitable for IoT scenarios.

我用 Python 做处理层，是因为它开发效率高，适合做业务逻辑、规则计算和数据处理。
I used Python for the processing layer because it has high development efficiency and is well suited for business logic, rule evaluation, and data processing.

我用 SQLite 作为默认数据库，是因为它轻量、部署简单，适合本地开发和演示。
I used SQLite as the default database because it is lightweight and easy to deploy, which is ideal for local development and demos.

同时我也预留了 MySQL 扩展能力，说明这个系统具备从演示环境向更正式环境演进的空间。
At the same time, I also reserved MySQL extensibility, which shows that the system has room to evolve from a demo environment toward a more formal environment.

## 8. 项目难点 / Project Challenges

我觉得这个项目主要有三个难点。
I think this project had three main challenges.

第一个难点，是让 C++ 和 Python 两端通过 MQTT 稳定协同，包括 Broker 配置、消息格式统一和客户端连接问题。
The first challenge was making the C++ and Python sides work reliably together through MQTT, including broker configuration, unified message format, and client connection issues.

第二个难点，是把 Dashboard 从简单的数据展示页面，升级成可以支持告警和维护闭环的操作界面。
The second challenge was upgrading the dashboard from a simple data display page into an operational interface that supports a full alert and maintenance loop.

第三个难点，是既要保证本地环境容易跑通，又要保留扩展到真实链路的能力。
The third challenge was balancing easy local setup with the ability to extend into a more realistic production-style path.

为了解决这个问题，我做了 sample 模式、SQLite 优先方案，以及原生 MQTT 和 fallback 的双路径设计。
To solve this, I implemented a sample mode, a SQLite-first strategy, and a dual-path design with both native MQTT and a fallback mechanism.

## 14. 高频追问速答 / Quick Answers for Common Follow-Up Questions

### 1) 为什么用 MQTT？ / Why MQTT?

因为 MQTT 轻量、实时、解耦，非常适合设备到后端之间的消息传输。
Because MQTT is lightweight, real-time, and decoupled, which makes it very suitable for message transport between devices and the backend.

### 2) 为什么用 C++ 和 Python 分工？ / Why split C++ and Python?

因为 C++ 更适合模拟边缘设备和采集侧场景，而 Python 更适合快速实现处理逻辑和数据分析。
Because C++ is better suited for edge-device and collection-side scenarios, while Python is better for quickly implementing processing logic and data analysis.

### 3) 这个项目的核心价值是什么？ / What is the core value of this project?

核心价值是把工业设备监控从单点采集升级成一个完整的数据闭环和运维闭环。
The core value is that it upgrades industrial device monitoring from isolated data collection into a complete data loop and operations loop.

### 4) 如果继续扩展，你会做什么？ / If you keep extending it, what would you do?

我会接入更真实的设备数据源，增加更复杂的预测性维护模型，并加强告警与维护的时间线管理。
I would connect more realistic device data sources, add more advanced predictive-maintenance models, and strengthen the timeline management of alerts and maintenance.
